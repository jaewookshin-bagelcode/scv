# review

작업 중인 변경을 커밋 전에 점검한다. **읽기 위주**(read/grep/git diff)로 보고, 함부로 고치지
않는다 — 발견을 보고하고 수정은 사용자 확인 후.

## 절차
1. 범위 파악: `git diff`(+ `git diff --staged`)와 `git status` 로 바뀐 파일·라인을 본다.
   필요하면 `read`/`grep` 으로 주변 맥락을 확인한다.
2. 각 변경을 아래 관점으로 본다:
   - **정확성/버그**: 경계조건, off-by-one, None/에러 처리, 비동기 경쟁, 자원 누수.
   - **보안**: 신뢰 불가 입력, 경로 탈출, 비밀 노출, 명령/SQL 주입.
   - **설계/일관성**: 중복, 네이밍, 주변 코드 관례와의 일치, 죽은 코드.
   - **테스트**: 새 동작에 테스트가 있는가. 문서/주석이 동작과 일치하는가.
3. 보고: 발견을 **심각도(blocker/major/minor/nit)** 와 `파일:라인` 으로 정리한다. 추측은
   추측이라 밝히고, 확실한 것과 구분한다.

## 규칙
- 리뷰 단계에선 코드를 직접 수정하지 않는다(사용자가 고치라고 하면 그때). 거짓 통과 금지 —
  문제가 있으면 분명히 적는다.
